## 1. 项目背景与目标

### 1.1 背景

- 用户已开通 Cloudflare 域名与开发者账号,需要搭建一个**多服务商图片生成 SaaS 平台**,首批接入 `gpt-image-2` 兼容协议厂商,后续扩展。
- 用户希望服务的所有基础设施运行在 Cloudflare,享受边缘网络、低运维、统一计费。

### 1.2 目标

1. **稳定提供文生图、图生图、连续对话三种生图模式**,支持图片大小、张数选择。
2. **3 级用户权限体系**(系统管理员 / 管理员 / 普通用户),支持配额管理、密钥分配、统计观测。
3. **图片资产私有化**:存于 Cloudflare R2,非授权链接不可访问;通过鉴权接口代理读取。
4. **异步生成体验**:用户提交后即可离开,回来仍能查看进度与结果;在线时实时推送状态变化。
5. **国际化**:中英文,亮/暗双主题。
6. **可扩展**:新增第三方服务商成本低;密钥与服务商解耦。

### 1.3 非目标(本期不做)

- 支付与订阅付费(自用 / 内部使用,管理员手动分配配额)。
- 移动 App(响应式 Web 即可,适配桌面 + 平板 + 手机)。
- 图片二次编辑(裁剪、滤镜)等深度后期。
- 多语言扩展超出中英,即使预留接口也只实现两语言。

## 2. 业务需求

### 2.1 角色与权限矩阵

| 功能                         |  普通用户  |      管理员      | 系统管理员 |
| ---------------------------- | :--------: | :--------------: | :--------: |
| 登录 / 修改密码 / 修改昵称   |     ✅     |        ✅        |     ✅     |
| 使用图片生成(三种模式)       | ✅(受配额) |    ✅(受配额)    | ✅(无配额) |
| 查看本人会话/历史(分页)      |     ✅     |        ✅        |     ✅     |
| 创建普通用户并分配配额       |     ❌     |  ✅(配额≤自身)   |     ✅     |
| 查看下属用户的"用量统计"     |     ❌     | ✅(仅本人创建的) |  ✅(全部)  |
| 查看下属用户的"具体生图内容" |     ❌     |        ❌        |     ✅     |
| 创建管理员                   |     ❌     |        ❌        |     ✅     |
| 给管理员分配密钥             |     ❌     |        ❌        |     ✅     |
| 配置服务商 / 服务商密钥      |     ❌     |        ❌        |     ✅     |
| 切换自己使用哪个服务商/密钥  |     ❌     |        ❌        |     ✅     |
| 全局统计仪表盘               |     ❌     |        ❌        |     ✅     |

### 2.2 配额规则

- **计费单位**:**1 次生图请求 = 1 个配额**(无论成功或失败)。
- **管理员分配下属配额**:不能超过自身剩余配额(扣减:`admin.remaining -= delta`)。
- **管理员的总配额**:由系统管理员分配,管理员看到的"自身配额 = 已分配 - 已用"。
- **系统管理员**:无配额限制(`is_unlimited = true`)。
- **额度消耗时机**:`POST /api/generate` 创建任务时**预占用 1 次**;若任务最终系统侧失败(不是API失败),回退。API 失败仍计费。
- **额度续期**:系统管理员可手动给管理员"加配额";管理员可手动给下属"加配额"。无周期自动续期(MVP)。

### 2.3 核心业务流

```
[创建用户]
sysadmin → 创建 admin → 指定密钥 → 分配总配额
admin → 创建普通用户 → 分配配额(≤自身剩余) → 默认继承 admin 的密钥

[生图]
user 提交 prompt → 后端鉴权 + 配额校验 + 选定密钥 → 入队任务
→ 异步调用第三方 API → 存图到 R2 → 写元数据到 D1 → 推送状态(WebSocket/SSE)
→ 用户在线: 实时见结果; 离线: 下次进入直接看历史

[查看图片]
任意页面渲染图片 → 走 /api/i/:imageId → 后端校验 JWT + 权限 → R2.get → 返回(带短期Cache)
```

### 2.4 关键 UX 行为

- **离开-回来体验**:任务态由后端持久化,前端首屏加载会话/历史时获得最新状态(已完成 / 处理中 / 失败)。
- **在线实时推送**:WebSocket 连接到任务的 Durable Object,接收状态/进度变化。
- **历史分页**:倒序按时间分页,默认 20 条/页,支持向上加载更早消息(连续对话)。
- **大图查看**:点击图片打开大图模态,支持原图下载、复制 prompt、删除消息。
- **失败重试**:失败的任务有"重试"按钮,基于原参数发起新任务(占用新配额)。

## 3. 第三方 API 协议分析 (gpt-image-2)

> 来源: [`docs/references/gpt-image2工具.html`](./references/gpt-image2工具.html)
>
> 该服务商为 **OpenAI 兼容协议**,兼容 Responses API 与 Legacy Image/Chat API。

### 3.1 端点与协议

#### 文生图 (Text-to-Image)

- **主端点(Responses API)**: `POST {BASE_URL}/v1/responses`
  ```json
  {
    "model": "gpt-image-2",
    "input": "<prompt 字符串>",
    "tools": [{ "type": "image_generation", "size": "1024x1024" }]
  }
  ```
- **回退端点(Legacy)**: `POST {BASE_URL}/v1/images/generations`
  ```json
  { "model": "gpt-image-2", "prompt": "...", "n": 1, "size": "1024x1024" }
  ```

#### 图生图 (Image-to-Image / Edit)

- **主端点(Responses API)**: `POST {BASE_URL}/v1/responses`
  ```json
  {
    "model": "gpt-image-2",
    "input": [
      {
        "role": "user",
        "content": [
          { "type": "input_text", "text": "<prompt + 编辑指令>" },
          { "type": "input_image", "image_url": "data:image/png;base64,<...>" }
        ]
      }
    ],
    "tools": [{ "type": "image_generation", "size": "1024x1024" }]
  }
  ```
- **回退端点(Legacy chat/completions)**: `POST {BASE_URL}/v1/chat/completions`
  ```json
  {
    "model": "gpt-image-2",
    "messages": [
      {
        "role": "user",
        "content": [
          { "type": "text", "text": "..." },
          { "type": "image_url", "image_url": { "url": "data:image/png;base64,..." } }
        ]
      }
    ]
  }
  ```
- **Mask(涂抹局部编辑)**:工具不支持原生 mask 字段,采用"原图 + 半透明红色叠加图"两张图作为视觉指令(在 prompt 中说明:仅修改红色覆盖区域)。

#### 连续对话 (Chat)

- **端点**: `POST {BASE_URL}/v1/chat/completions`
  ```json
  {
    "model": "gpt-image-2",
    "messages": [
      { "role": "user", "content": "..." },
      { "role": "assistant", "content": "..." },
      { "role": "user", "content": "..." }
    ]
  }
  ```

### 3.2 鉴权与请求头

```
Authorization: Bearer <API_KEY>
Content-Type: application/json
Accept: application/json
```

### 3.3 参数

| 参数  | 取值                                                                                                                      | 说明                                                                                                   |
| ----- | ------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| model | `gpt-image-2`                                                                                                             | 模型 ID;不同服务商可能不同                                                                             |
| size  | `1024x1024` / `1024x1536` / `1536x1024` / `2048x2048` / `2880x2880` / `3840x2160` / `2160x3840` / `auto` / 自定义(16倍数) | 输出尺寸                                                                                               |
| n     | 1+                                                                                                                        | 张数;**`image_generation` tool 不接受 n**;实际由客户端循环 N 次单图请求(每次 n=1)以保证"扣几次=出几张" |

### 3.4 响应解析

服务商响应可能是以下几种结构之一,系统需健壮兼容:

1. **Responses API**:
   ```json
   {
     "output": [
       { "type": "image_generation_call", "result": "<base64>" },
       { "type": "message", "content": [{ "type": "output_image", "image_url": "..." }] }
     ]
   }
   ```
2. **Legacy Images**: `{ "data": [ { "url": "..." } | { "b64_json": "..." } ] }`
3. **Chat Completions**: `{ "choices": [ { "message": { "content": "...", "images": [...] } } ] }`,内容可能含 `<img src=>` HTML、Markdown 图片、裸 URL、内嵌 base64。

### 3.5 关键工程注意点

- **请求体大小**:base64 图片可能让请求体超 4MB;部分代理拒收 → 后端自行编码,避免冗余 + 必要时调整尺寸。Cloudflare Workers 请求体上限 100MB(Free/Pro),足够;但出站要尊重对方代理上限。
- **静默降级**:部分代理会把请求的高分辨率(如 4K)默默降到 1.57M 像素;需对比"请求 size vs 实际返回 size",必要时记日志告警。
- **超时**:gpt-image-2 单次响应可能 30–120s;Workers 单 fetch 子请求空闲超时约 100s — **不允许前端 Worker 同步等结果**,改走异步任务编排。
- **失败回退**:若主端点返回 `404/405/501`,自动尝试回退端点(Legacy)。
- **批处理**:多参考图场景,后端可拆为 N 个独立请求并发(上限 5)。

### 3.6 抽象层设计

为支持后续多服务商,需要抽象 `ImageProvider` 接口:

```ts
interface ImageProvider {
  id: string; // 服务商唯一 ID
  name: string; // 显示名
  generate(req: GenerateRequest): Promise<GenerateResponse>;
  // 内部封装鉴权、端点选择、响应规整、错误标准化
}
```

首批实现 `OpenAICompatibleProvider`(覆盖 gpt-image-2)。

### 4.3 关键架构决策

#### D-1. 长任务编排:**Workflows + Durable Object**

- `POST /api/generate` 入口 Worker:鉴权 + 配额校验 + 创建 task 行(`status=queued`)+ 启动 Workflow 实例 + 创建/获取对应 `TaskRoom` DO + 立即返回 `{ taskId, wsUrl }`。
- Workflow 步骤:
  1. `step.do('callImageAPI')` — 调第三方 API(可重试,有指数退避)。
  2. `step.do('uploadR2')` — 解码 base64,上传 R2;若有多张并行 upload。
  3. `step.do('updateD1')` — 写图片元数据。
  4. `step.do('notifyDO')` — RPC 通知 DO 推送给所有连着的 WS 客户端。
- Workflow 失败:每步独立重试;最终失败写 D1 并通知 DO。

#### D-2. 实时进度:**Durable Object + WebSocket Hibernation**

- 每个生成任务对应一个 DO 实例(`taskRoom_${taskId}`)。
- Worker 升级 WebSocket 后通过 DO RPC 把 socket 接入。
- DO 用 **Hibernation API** (`acceptWebSocket`):闲置不计费,有消息时唤醒。
- 状态变化时,Workflow → 通过 DO RPC `broadcast(progress)`。
- DO 内部用 `state.storage` 持久化最新状态,新连接到来时立即 push 最新值(防错过)。

#### D-3. 图片访问鉴权:**Worker 代理 fetch R2 + JWT**

- 图片 URL: `/api/i/:imageId?t=<short-lived-jwt>` 或 cookie 模式。
- Worker:
  1. 校验 JWT(签发短期 token,15min 有效)。
  2. 查 D1 `image_objects` 表,获取 R2 key + owner + 权限。
  3. 校验"当前用户是否有权限访问"(本人 / 系统管理员)。
  4. `env.R2.get(key)` → 返回流。
  5. 写 Cache API(短期 5min,Vary 含 Authorization),命中下次直接返回。
- **不**使用 Pre-signed URL(无法实时撤权 / 加配额)。

#### D-4. 多服务商 + 多密钥

- 表 `providers`:`id, name, base_url, default_model, enabled, ...`。
- 表 `provider_keys`:`id, provider_id, label, encrypted_key, owner_admin_id, allocated_quota, used_quota, enabled`。
- 加密:用 Worker Secrets 中的 `KEY_ENCRYPTION_KEY` 通过 Web Crypto AES-GCM 加密 → D1 存密文。运行时即用即解。
- 系统管理员创建管理员时,从 `provider_keys` 选一条赋给该 admin;admin 创建普通用户时,继承自己的密钥。
- 系统管理员自身可在"个人偏好"切换"默认服务商 / 密钥"。

#### D-5. ORM 与迁移

- **Drizzle ORM** + `drizzle-kit generate` + `wrangler d1 migrations apply`。
- 单一 schema 仓库(`/server/db/schema.ts`),前端不直接 import 但可 import 类型(`InferSelectModel`)。

#### D-6. 部署形态:**单 Worker 同时托管 SPA + API**

- 一个 Worker 项目:`wrangler.jsonc` 包含 `assets`, `routes`, `bindings`(D1, KV, R2, DO, Workflow, AI Gateway, Secrets)。
- `assets.run_worker_first: ["/api/*", "/ws/*"]` — Worker 先于静态资源处理 API/WS。
- 部署:`wrangler deploy --env=staging|production`。

// quota_transactions
id (pk), user_id (fk), delta (signed int), reason (enum: 'admin_grant'|'task_charge'|'task_refund'),
operator_id (fk users.id), task_id (fk, null), created_at

// sessions
id (uuid pk), user_id (fk), title, mode (enum: 'text2image'|'image2image'|'chat'),

// messages

attachments (json), task_id (fk, null=纯对话消息), created_at

// tasks
id (uuid pk), session_id (fk), message_id (fk), user_id (fk), provider_key_id (fk),
status (enum: 'queued'|'running'|'succeeded'|'failed'|'cancelled'),
mode, params (json: {size,n,model,...}), error_code, error_msg,
provider_request_id (string, null), provider_raw_response (blob/text, redacted),
queued_at, started_at, finished_at, retry_of (fk tasks.id, null)

// image_objects
id (uuid pk), task_id (fk), session_id (fk), owner_user_id (fk),
r2_key (string uniq), mime, width, height, byte_size,
sha256 (uniq for dedup), is_reference (bool, true=用户上传作参考图),
created_at, deleted_at (soft delete)

// audit_logs
id (pk), actor_id (fk), action (enum), target_type, target_id, payload (json), created_at, ip

## 6. 服务端 API 设计

### 6.1 总体规范

- 所有 API 前缀:`/api/`
- WebSocket:`/ws/task/:taskId`
- 鉴权:`Authorization: Bearer <jwt>` 或 HTTP-only Cookie(MVP 推荐 Cookie + CSRF token)
- 时间戳全部毫秒
- 错误格式:
  ```json
  { "error": { "code": "QUOTA_EXCEEDED", "message": "...", "details": {} } }
  ```

## 8. UI/UX 设计风格

### 8.1 候选设计系统

参考 [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md),69 个设计系统中,最契合"AI 画图 SaaS / 工作台 + 聊天"的候选:

| #   | 风格                   | 关键词                                | 适合                     |
| --- | ---------------------- | ------------------------------------- | ------------------------ |
| 1   | **Linear**             | 极简、克制、灰阶 + 紫色强调、流畅过渡 | 工作台、列表、表格       |
| 2   | **Vercel**             | 黑白对比、纯净、无衬线英文字体        | 仪表盘、设置、文档化页面 |
| 3   | **Anthropic / Claude** | 暖橙强调、温和无压迫、注重可读        | AI 聊天体验、prompt 输入 |
| 4   | **Notion**             | 留白、模块化、亲和                    | 历史/会话归档            |
| 5   | **Raycast**            | 命令式、高密度信息、暗色优先          | 高级用户设置             |

### 8.2 最终选定:**Linear + Anthropic 混搭(Linear 主调,聊天区借鉴 Anthropic 暖色)**

**理由**:

- Linear 的极简灰阶 + 高对比度 + 圆角较小的卡片,适合工作台/管理员仪表盘——保证大量数据和操作不显凌乱。
- 聊天区 / 输入区借鉴 Anthropic Claude 的暖橙(#D97706 / #F59E0B 系)作为主品牌色,营造"创作而非冷冰冰任务"的氛围。
- 二者都有成熟的暗色主题。

**色板**:

| Token                        | Light                   | Dark                    |
| ---------------------------- | ----------------------- | ----------------------- |
| `--background`               | `oklch(0.99 0 0)`       | `oklch(0.16 0.005 240)` |

| `--card`                     | `oklch(1 0 0)`          | `oklch(0.19 0.005 240)` |
| `--border`                   | `oklch(0.92 0 0)`       | `oklch(0.27 0.005 240)` |
| `--primary` (Anthropic 暖橙) | `oklch(0.71 0.15 60)`   | `oklch(0.78 0.15 60)`   |
| `--primary-foreground`       | `oklch(0.18 0 0)`       | `oklch(0.18 0 0)`       |

**阴影**:仅用于浮层(`shadow-lg`,带 `ring-1 ring-black/5`),普通卡片靠 border 区分。

**字体**:

- ASCII: `Inter` / `Geist` (回退 system-ui) — Linear/Vercel 风格。
- CJK: `"PingFang SC", "Hiragino Sans GB", "Microsoft YaHei", sans-serif`。
- 等宽:`"JetBrains Mono"`(prompt 调试、代码块)。

### 8.3 关键交互

- 主按钮:实色暖橙背景 + 白字,hover 提亮 4-6%。
- 次按钮:白底/暗底 + 边框 + 灰字。

## 10. 安全设计

### 10.1 身份与会话

- 密码:`bcrypt`(cost=12)或 `argon2id` (Workers 上更现代选 argon2id - via `@noble/hashes` 等纯 JS 包)
- JWT:HS256,签发 access(15min) + refresh(7d);refresh 存 httpOnly cookie
- 登出:KV 写入 jti 黑名单(到 access 过期才能删)
- CSRF:对于 cookie 模式 API,所有非 GET 路径要求 `X-CSRF-Token` 头,值与 cookie 中独立的 csrf cookie 匹配
- Turnstile:登录、注册、忘记密码三处必上;Worker 端校验 token

### 10.2 密钥与机密

- `KEY_ENCRYPTION_KEY` 走 `wrangler secret put`(Workers Secrets,环境变量级)
- `provider_keys.encrypted_key` = AES-GCM(明文密钥, 主密钥, IV) → 入库 base64
- 解密只在调用前一刻进行,调用后内存清零
- 系统管理员页面只显示密钥末 4 位(`abcd`),不回显完整密钥

### 10.3 图片与对象访问

- R2 桶:**不**绑定公开域名;仅 Worker 内 binding 访问
- 图片 URL 短期 token(15min);跨设备打开历史时通过登录态再换 token
- 防猜测:imageId 用 ULID/UUIDv7

### 10.4 注入与校验

- 所有入参用 `zod` / `valibot` 校验(Hono 内置 `@hono/zod-validator`)
- prompt 长度上限 4000 字符(防滥用)
- 上传图片:MIME 白名单、单文件 ≤ 10MB、批量 ≤ 5 张;经过服务端再压缩到 ≤2048 边
- SQL 全部用 Drizzle 参数化,不拼接

### 10.5 速率限制

- IP+UA 维度:登录 5次/15分(Cloudflare Rate Limiting Rules)
- 用户维度:生图 60次/分钟(防误触并发,强制串行也用限流)

### 10.6 审计

- `audit_logs`:任何写操作(创建用户、改配额、改密钥、改服务商、登录登出)入库
- 系统管理员可查询/导出

## 14. 风险与未决项

### 14.1 待确认

- [ ] **Cloudflare Secrets Store** GA 时间(目前 open beta);MVP 用 Workers Secrets + D1 加密替代,GA 后切。
- [ ] **AI Gateway** 是否对所有第三方 base URL 透明代理(本身要求按 provider 配置;`gpt-image-2` 这类**自定义代理**接入方式需测试)。如果不支持,需把"调用日志"自行 fanout 到 Workers Logs/D1。
- [ ] **gpt-image-2 单次请求最长耗时**:用真实账号跑 95 分位,据此调整超时与重试。
- [ ] **R2 Pre-signed URL 替代方案考量**:目前选 Worker 代理;若图片量大(>10K/日)需评估边缘计算成本。
- [ ] **邮件**:Resend vs MailChannels Worker;MailChannels Worker 在 Cloudflare 内免费但配置 SPF/DKIM 较繁琐。

### 14.2 已知风险

1. **Workers 单次 fetch 100s 空闲超时** — 第三方若 >100s 必须改 polling;本期通过 Workflow `step.do + step.sleep` 化解。
2. **D1 单库 10GB** — 长期增长后需评估分库或归档老数据(R2 + 索引)。
3. **WebSocket Hibernation 序列化限制 2048 字节** — 任务态精简存储,大对象走 `state.storage`。
4. **Turnstile 在中国地区**:Cloudflare 一些服务在中国不可用;若目标用户主要在中国,需评估替代方案。

### 14.3 未来扩展

- 多语言扩展(日韩等)
- 移动端 App(基于 Capacitor 重用 Web 代码)
- 服务商 webhook 模式(替代轮询 / 长连接)
- 团队 / 工作区(多用户协作)
- 套餐与支付(Stripe)
- API 开放(对外 API + OAuth Provider)
